## 서론

요즘의 개발은 불과 6~7년 전과도 너무나 다르게 변했습니다. AI 도구들이 생기고, 여러 자동화 도구, 프레임워크가 빠르게 발전하면서, 개발자들이 마주하는 문제와 그 문제를 해결하는 방식도 크게 달라졌습니다.

저는 이 시점에서 과거에는 어떻게 문제를 해결해왔는지, 그리고 지금은 어떻게 달라졌는지를 되돌아보고 싶었습니다.

아래 글은 제가 지금까지 겪었던 일들을 되돌아보며, 과거의 방식과 현재의 방식을 비교한 글입니다.

## 본론

### CI/CD 파이프라인을 처음 구축할 때

**2019–2020** · Vue + Electron 기반 서비스의 배포 인프라를 처음부터 구성

<Compare>
<Then label="라떼는" icon="🕰️">

첫 회사에 입사했을 때 저는 Vue와 JavaScript 경험이 있었지만, 배포 인프라는 거의 모르는 상태였습니다. 그런데 "서비스 개발하면서 배포도 알아서 해줘"라는 역할이 주어졌습니다. AWS 콘솔을 처음 열던 날의 막막함을 아직도 기억합니다.

<StepFlow>
<Step number={1} title="AWS S3 구조 이해">

"정적 파일을 어디에 올리고, CDN을 어떻게 앞에 붙이는지"를 공식 문서를 읽으면서 파악했는데, 용어가 낯설고 설명이 추상적이라 실제로 어떻게 동작하는지 감이 오기까지 시간이 걸렸습니다. 설정 하나를 잘못 건드려서 배포된 서비스가 접근 불가 상태가 된 경험도 있습니다.

</Step>
<Step number={2} title="AWS Amplify 도입">

GraphQL과 백엔드 없이 개발하는 방법으로 AppSync를 채택했는데, Amplify의 설정 방식, GraphQL 스키마 정의, 인증 설정이 일반적인 REST API와는 완전히 달랐습니다. 공식 문서와 GitHub 이슈를 번갈아 보면서, 왜 이 에러가 나는지 파악하는 데만 하루가 넘게 걸리는 경우도 있었습니다.

</Step>
<Step number={3} title="Electron 데스크탑 앱 빌드">

Windows 설치 파일을 만들기 위한 패키징 설정, 자동 업데이트 메커니즘, 코드 서명 — 이 모든 것들을 스스로 찾아가며 구성했습니다. 특히 사내 공용 PC에 배포해야 한다는 조건 때문에, 설치 방식과 업데이트 정책도 직접 설계해야 했습니다.

</Step>
</StepFlow>

지금 생각하면 그때의 시행착오가 나중에 Bitbucket Pipeline 구성이나 Firebase App Distribution 도입을 할 때 큰 도움이 됐습니다. 하지만 처음 배우는 과정은 정말 막막하고 길었습니다.

</Then>
<Now label="지금은" icon="🤖">

배포 인프라를 처음 구성하는 사람에게 가장 어려운 건 "전체 그림"이 없는 상태에서 작은 조각들을 맞춰나가야 한다는 점입니다. 지금이라면 이 문제를 다르게 접근합니다.

<StepFlow>

</Step>
<Step number={2} title="Amplify 설정 및 트러블슈팅">

</Step>
<Step number={3} title="파이프라인 YAML 초안 생성">

"Vue 앱의 빌드, 테스트, S3 배포까지 포함한 Bitbucket Pipeline 설정을 작성해줘. 브랜치별로 스테이징/프로덕션 배포를 구분해서." 파이프라인 설정 파일의 초안을 빠르게 얻고, 상황에 맞게 수정하는 방식으로 진행할 수 있습니다.

</Step>
</StepFlow>

처음 인프라를 구성하는 경험 자체가 주는 학습은 여전히 소중합니다. 하지만 '오래 헤매는 것'과 '제대로 이해하는 것'은 다릅니다. 지금이라면 먼저 전체를 이해한 뒤 구현에 들어갈 수 있습니다.

</Now>
</Compare>

### 성능 개선 작업을 할 때

**2021–2022** · 담당 서비스 단위 액션의 평균 응답 시간을 300ms 이하로 개선

<Compare>
<Then label="라떼는" icon="🕰️">

"이 버튼 누르면 왜 이렇게 느려요?" 라는 제보가 들어오는 순간부터 성능 개선이 시작됩니다. 당시 저는 담당 서비스에서 다양한 비즈니스 액션들을 개발하고 있었는데, 특정 작업들이 유난히 느리다는 피드백이 반복해서 들어왔습니다.

<StepFlow>
<Step number={1} title="병목 탐색">

브라우저 개발자 도구의 Network 탭을 열어보면 API 응답 시간이 보이지만, 어느 API가 느린지, API 내부에서 어디가 느린지는 보이지 않습니다. 프론트엔드 렌더링 문제인지, 백엔드 쿼리 문제인지를 구분하려면 양쪽 로그를 비교해야 했습니다. 당시에는 체계적인 성능 프로파일링 기준도 없었고, 어디서부터 봐야 하는지 감을 잡는 데 시간이 걸렸습니다.

</Step>
<Step number={2} title="백엔드와의 협업">

"이 API가 느린데 봐줄 수 있어요?"라고 막연하게 요청하면, 상대도 어디를 봐야 할지 모릅니다. 재현 조건, 느린 정도, 어떤 상황에서 발생하는지를 구체적으로 정리해서 전달해야 했는데, 이런 협업 방식이 정착되기까지 시간이 걸렸습니다. 서로 다른 로그 형식과 모니터링 도구를 쓰다 보니, 같은 문제를 보는 시각 자체가 달랐습니다.

</Step>
</StepFlow>

이런 과정들을 거쳐서 결국 단위 액션의 평균 응답 시간을 300ms 이하로 줄였습니다. 결과는 좋았지만, 그 과정에서 병목 탐색에 쓴 시간이 실제 수정 시간보다 훨씬 길었습니다.

</Then>
<Now label="지금은" icon="🤖">

성능 개선 작업에서 달라지는 건 주로 "문제를 정의하는 속도"입니다.

<StepFlow>
<Step number={1} title="AI로 병목 분석">

막연하게 느리다는 피드백이 들어왔을 때, 지금이라면 먼저 분석부터 시작합니다. "이 API 호출 흐름을 분석해줘. 네트워크 요청 수, 렌더링 사이클, 불필요한 리렌더링이 있는지 확인해줘." 이런 요청으로 브라우저 개발자 도구를 혼자 뒤지던 시간을 줄일 수 있습니다.

</Step>
<Step number={2} title="협업 요청서 자동화">

"이 API가 평균 450ms가 걸리는데, 프론트엔드에서 줄일 수 있는 부분과 백엔드에서 봐야 할 부분을 구분해서 정리해줘. 재현 조건도 같이 정리해줘." 이렇게 협업 요청서 수준의 내용을 빠르게 만들어낼 수 있습니다. 상대방이 어디를 봐야 하는지를 먼저 정리해서 전달하면 협업의 효율이 달라집니다.

</Step>
<Step number={3} title="사전 알람으로 전환">

"이 Datadog 로그 패턴에서 성능 이상을 감지하는 알람 조건을 어떻게 설정하면 좋을지 제안해줘." 사후 대응이 아닌 사전 감지로 접근 방식을 바꿀 수 있습니다.

</Step>
</StepFlow>

핵심은 병목을 찾는 데 드는 시간이 줄어든다는 겁니다. 무엇이 문제인지 이해하는 데 쓰던 시간을, 실제 수정과 검증에 쓸 수 있습니다.

</Now>
</Compare>

---

성능 문제가 어느 정도 잡혀갈 무렵, 이번엔 완전히 처음 보는 영역으로 뛰어들어야 하는 상황이 왔습니다.

### 모르는 기술을 새롭게 도입할 때

**2023** · iOS/Android WebView 앱을 Flutter로 전환

<Compare>
<Then label="라떼는" icon="🕰️">

2023년 당시 저는 React와 TypeScript를 주로 다루고 있었습니다. 그런데 어느 날 기존 WebView 기반의 iOS/Android 앱을 하나의 코드베이스로 전환해야 한다는 요청을 받았습니다. "iOS, Android 각각 WebView 앱들을 유지보수하기 힘드니, Flutter 혹은 React Native로 다시 만들어보자." — 저한테는 Flutter도, React Native도 이름만 들어봤을 뿐이었습니다.

<StepFlow>
<Step number={1} title="기존 코드 파악">

Swift와 Kotlin을 처음 보는 입장에서, 기존 앱의 화면 흐름과 기능을 분석하려면 문서도 없고, 물어볼 사람도 마땅치 않았습니다. 거의 매일 스택 오버플로우와 공식 문서를 번갈아 보면서, iOS에서 `UINavigationController`가 왜 이렇게 동작하는지, Kotlin에서 `CoroutineScope`가 뭔지를 이해하는 데만 일주일이 넘게 걸렸습니다.

</Step>
<Step number={2} title="Flutter vs React Native 선택">

팀원들과 함께 어떤 기술을 선택할지 논의했는데, 기준 자체를 잡는 것부터 어려웠습니다. 두 기술 모두 직접 써본 사람이 없었기 때문입니다. 결국 각자가 간단한 PoC를 만들어보고 직접 비교했습니다. Flutter를 선택한 결정적인 이유는 더 매끄러운 렌더링 성능과 Dart 언어의 학습 용이성이었습니다. React Native는 JavaScript 친화적이었지만, 당시 Bridge 방식의 성능 이슈가 있었고 Expo의 제약도 고려 대상이었습니다.

</Step>
<Step number={3} title="Flutter 자체 학습">

Widget 트리 구조부터, StatefulWidget과 StatelessWidget의 차이, Provider를 쓸 건지 Riverpod를 쓸 건지 — 상태 관리 라이브러리 선택만 해도 적지 않게 고민했습니다. 공식 문서를 읽고, 유튜브 강의를 보고, 예제를 따라 치면서 익혔습니다.

</Step>
<Step number={4} title="iOS App Store 배포">

Flutter 앱을 처음 App Store에 올리는 과정에서 인증서, 프로비저닝 프로파일, Xcode 설정까지 건드려야 했는데, 앱 개발 경험이 없던 저한테는 이것도 처음 보는 영역이었습니다. 빌드가 안 되는 이유를 찾아가며 Xcode 오류 메시지를 구글에 검색하는 일을 하루에도 수십 번 반복했습니다.

</Step>
</StepFlow>

이런 과정들을 거쳐 결국 Flutter로 앱을 완성했습니다. 하지만 이 모든 과정에 약 3개월이 소요되었습니다. 기술 선택과 분석에 한 달, 실제 개발에 두 달. 지금 돌이켜보면 그 3개월의 대부분은 "몰라서 헤매는 시간"이었습니다.

</Then>
<Now label="지금은" icon="🤖">

지금 같은 상황이 다시 온다면 어떻게 할까. 솔직히 말하면, 3개월이 아닌 3주 안에 끝낼 수 있을 것 같습니다.

<StepFlow>
<Step number={1} title="기존 앱 구조 파악">

당시 저는 Swift와 Kotlin 코드를 읽는 데 일주일 이상을 썼습니다. 지금이라면 기존 앱의 소스 코드를 Claude Code에 넘기고 먼저 물어볼 겁니다. "이 iOS 앱의 화면 흐름과 주요 기능을 정리해줘. API 통신 방식, 상태 관리, Firebase 연동 부분도 함께 파악해줘." 코드를 전부 직접 읽지 않아도, 앱의 전체 구조와 각 화면이 어떤 역할을 하는지를 빠르게 파악할 수 있습니다.

</Step>
<Step number={2} title="기술 선택 트레이드오프 대화">

당시엔 각자 PoC를 만들면서 직접 비교했는데, 지금이라면 먼저 상황을 설명하고 트레이드오프를 물어볼 겁니다. "소규모 팀으로 iOS/Android 앱을 만드는데, 우리 팀이 React/TypeScript 경험이 있다는 점도 고려해서 Flutter와 React Native 중 어떤 게 나은지 비교해줘." 각 기술의 제약이나 커뮤니티 동향, 실제 사용 시 만날 수 있는 문제들도 함께 이해할 수 있습니다.

</Step>
<Step number={3} title="탐색 → 이해 → 구현 순서 유지">

"Flutter에서 Provider 대신 Riverpod을 쓰는 게 왜 나은지 설명해줘"처럼 개념부터 먼저 물어보고, 이해한 다음에 구현을 맡깁니다. 이해하지 못하는 코드를 그냥 가져다 쓰면, 나중에 문제가 생겼을 때 어디를 봐야 하는지 모르는 상황으로 이어집니다.

</Step>
</StepFlow>

3개월이 3주가 된다고 했지만, 중요한 건 단순히 빠른 것만이 아닙니다. 지금이라면 탐색 → 이해 → 구현의 순서를 지키면서, 실제로 내가 짠 코드처럼 이해하면서 진행할 수 있습니다.

</Now>
</Compare>

---

그로부터 1년이 지나, 이번엔 새로운 기술을 익히는 것이 아니라 오래된 것들을 정리하는 작업이 기다리고 있었습니다.

### 레거시 시스템을 마이그레이션할 때

**2024** · PHP(CodeIgniter) 레거시를 Next.js로 점진적 전환

<Compare>
<Then label="라떼는" icon="🕰️">

합류한 프로젝트에서 가장 먼저 마주한 건 PHP(CodeIgniter)로 쌓인 레거시 코드였습니다. 전임자들이 수년에 걸쳐 작성한 코드는 문서도 거의 없고, 함수 하나가 수백 줄을 넘기는 경우도 있었습니다. Next.js 개발자로 합류했지만, 당장 레거시를 건드리지 않으면 새 기능을 추가하기가 어려운 상황이었습니다.

<StepFlow>
<Step number={1} title="인증 시스템 역추적">

CodeIgniter가 `ci_session`이라는 독자적인 세션 쿠키를 사용하고 있었는데, Next.js로 전환한 페이지가 이 세션을 인식하지 못하면 사용자가 로그인 상태를 유지할 수 없었습니다. PHP 세션의 직렬화 방식, 암호화 로직, 쿠키 구조를 직접 역추적해서 Next.js 환경에서 동일하게 동작하는 쿠키를 새로 설계해야 했습니다.

</Step>
<Step number={2} title="PHP 코드 분석">

변수 이름이 축약형이거나, 비즈니스 규칙이 SQL 쿼리 안에 인라인으로 박혀 있거나, 어떤 페이지에서 어떤 세션 키를 쓰는지 파악하려면 코드를 처음부터 끝까지 직접 읽어야 했습니다. 전환 대상 페이지를 분석하는 데만 며칠씩 걸렸습니다.

</Step>
<Step number={3} title="점진적 전환 검증">

PHP 페이지와 Next.js 페이지가 동시에 운영되면서 공존해야 했기 때문에, 한쪽을 바꾸면 다른 쪽에 영향을 주는 경우가 많았습니다. 검색 기능을 전환할 때는 기존 PHP 검색 로직의 결과와 Next.js로 재구현한 결과가 일치하는지 하나하나 검증해야 했습니다.

</Step>
</StepFlow>

결과적으로 인증, 검색 등 핵심 페이지들은 성공적으로 전환했지만, 이 과정에서 예상보다 훨씬 많은 시간이 사라졌습니다. "코드가 뭘 하는지 파악하는 데"만 전체 공수의 절반 가까이가 쓰였습니다.

</Then>
<Now label="지금은" icon="🤖">

같은 레거시 마이그레이션을 지금 다시 한다면 가장 먼저 하는 일이 달라집니다. 코드를 직접 읽기 전에, AI에게 분석을 먼저 맡깁니다.

<StepFlow>
<Step number={1} title="AI로 코드베이스 분석">

"이 PHP 파일들의 전체 흐름을 분석해줘. 어떤 라우트가 있고, 각 라우트에서 어떤 세션 키를 읽고 쓰는지, 인증 처리가 어떻게 이뤄지는지 정리해줘." 이런 요청 하나로 직접 며칠이 걸렸을 코드 파악을 몇 시간으로 단축할 수 있습니다.

</Step>
<Step number={2} title="세션 역추적 질문">

"이 세션 쿠키 직렬화 방식을 설명해줘, Next.js에서 동일하게 구현하려면 어떻게 해야 해?" CodeIgniter의 `ci_session` 역추적도 AI와의 대화로 구체적인 구현 방향을 빠르게 잡을 수 있습니다.

</Step>
<Step number={3} title="검증 자동화">

"이 PHP 함수와 동일한 동작을 하는 TypeScript 함수를 작성하고, 동일한 입력에서 같은 결과가 나오는지 검증하는 테스트도 같이 만들어줘." 수작업으로 하나하나 눈으로 확인하던 검증을 자동화할 수 있습니다.

</Step>
</StepFlow>

단, 지금이라도 변하지 않는 것은 있습니다. 인증 로직은 AI가 분석해주더라도 제가 직접 검토합니다. 탐색과 이해는 빨라지지만, 검증은 여전히 사람의 몫입니다.

</Now>

## 결론

6년이 넘는 시간 동안 개발을 해왔습니다. 첫 회사에서 처음으로 Vue와 Electron을 팀에 도입하던 때부터, 이후 회사에서 대규모 서비스를 운영하고 레거시를 마이그레이션하던 때까지.

돌이켜보면 그 시간의 많은 부분이 "몰라서 헤매는 시간"이었습니다. 새로운 기술의 공식 문서를 처음부터 끝까지 읽고, 스택 오버플로우를 뒤지고, 오류 메시지를 구글에 붙여넣고, 시행착오를 반복하면서 익혀나갔습니다. 그 헤매는 과정 자체가 학습이었고, 그때 부딪혔던 문제들이 지금의 판단력을 만들어줬다는 건 분명합니다.

그런데 솔직히 말하면, 지금은 그 헤매는 시간이 많이 줄었습니다. AI 도구 덕분에 "몰라서 막히는 상황"이 빠르게 해소됩니다. 모르는 기술을 빠르게 파악하고, 설계 방향을 잡고, 초안을 만들고, 검증 기준을 세우는 속도가 달라졌습니다.

하지만 한 가지 확신하는 것이 있습니다. 빠르게 만들어진 코드를 이해하지 못한 채로 쓰면, 결국 문제가 생겼을 때 어디를 봐야 하는지 모릅니다. 탐색 → 이해 → 구현의 순서는 AI를 쓸 때도 동일하게 유효합니다. 속도가 빨라진 것이지, 이해를 건너뛰는 것이 아닙니다.

그리고 몇 년 뒤에는, 지금 이 방식도 또 다른 '라떼는 이렇게 했었는데...'가 될 것입니다. 그렇게 계속 쌓여가는 게 성장이라고 생각합니다.
